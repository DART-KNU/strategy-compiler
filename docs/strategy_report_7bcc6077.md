## 3. 팩터 계산 파이프라인 — Node Graph

### 3.1 전체 DAG 구조

```
[field: net_income_parent]  ──→
                               [combine: div]  → roe  → [cs_op: winsorize]  → roe_w
[field: total_equity_parent] ──→                         → [cs_op: zscore]    → roe_z

[field: vol_20d]  ──────────────────────────────────────→ [cs_op: winsorize]  → vol_w
                                                         → [cs_op: zscore]    → vol_z
                                                         → [cs_op: neg]       → vol_neg

[combine: weighted_sum]  ←── inputs: [roe_z, vol_neg],  weights: [0.6, 0.4]  →  score
```

### 3.2 노드별 상세 설명

#### `field: net_income_parent` — 지배주주 순이익
- **소스**: `mart_fundamentals_asof_daily.net_income_parent`
- **단위**: 백만 원 (분기 누적)
- **PIT 처리**: `available_date` 기준 — 분기 공시 후 실제로 열람 가능한 시점 이후만 사용

#### `field: total_equity_parent` — 지배주주 자기자본
- **소스**: `mart_fundamentals_asof_daily.total_equity_parent`
- **단위**: 백만 원 (기말 잔액)

#### `combine: div` — ROE 파생 계산

$$\text{ROE}_i = \frac{\text{net\_income\_parent}_i}{\text{total\_equity\_parent}_i}$$

- DB에 직접 제공되는 ROE 컬럼은 없음 → `combine` 노드의 `div` 연산으로 런타임 파생
- 자기자본이 0 또는 음수인 **자본잠식 종목**은 NaN → `null_policy: drop`으로 자동 제거
- **금융적 의미**: 자본 1원당 창출한 순이익. 높을수록 자본 효율이 좋은 기업. 워런 버핏이 주요 선별 지표로 강조한 퀄리티 팩터의 대표 지표

#### `cs_op: winsorize` — 극단값 처리

$$x_{\text{win}} = \text{clip}(x,\ P_5(x),\ P_{95}(x))$$

- 상하위 5% 극단값을 5·95 퍼센타일 경계로 대체
- **금융적 의미**: ROE는 부채 레버리지·일회성 이익 등으로 분포가 비대칭적이며 극단 이상값이 빈번. winsorize 없이 z-score를 취하면 이상값 1~2개가 전체 순위를 교란함

#### `cs_op: zscore` — 횡단면 표준화

$$z_i = \frac{x_{\text{win},i} - \mu}{\sigma}$$

- 해당 리밸런싱일 유니버스 전체 기준으로 평균 0, 표준편차 1로 정규화
- **금융적 의미**: 절대값이 아닌 상대적 순위를 스코어로 변환. 시장 전체 ROE 수준이 높아지거나 낮아져도 팩터 신호는 일관되게 상대 순위를 반영

#### `field: vol_20d` — 20일 실현 변동성
- **소스**: `mart_feature_daily.vol_20d`
- **계산 방법**: 직전 20 거래일 adj_close 수익률의 표준편차 × √252 (연환산)
- **금융적 의미**: 단기 가격 변동성. 저변동성 이상현상(Low Volatility Anomaly) — 낮은 변동성 종목이 장기적으로 위험 대비 초과수익을 내는 현상은 Blitz & van Vliet(2007), Baker et al.(2011) 등 다수 연구로 검증됨

#### `cs_op: neg` — 부호 반전

$$\text{vol\_neg}_i = -\text{vol\_z}_i$$

- `vol_20d`는 **낮을수록 유리**하므로 z-score 후 부호를 반전시켜 "높을수록 좋은" 방향으로 전환
- **주의**: 이 처리를 생략하면 고변동성 종목을 선호하는 치명적 방향 오류 발생

#### `combine: weighted_sum` — 복합 팩터 합성

$$\text{score}_i = 0.60 \times \text{roe\_z}_i + 0.40 \times \text{vol\_neg}_i$$

- **가중치 설계 논리**: ROE(퀄리티)를 주도 팩터로, 저변동성(방어)을 보완 팩터로 배분
  - ROE 60%: 수익성 높은 기업 선호
  - 저변동성 40%: 포트폴리오 변동성 억제 + Low Vol 프리미엄 포착

## 5. 이중 주기 점진적 리밸런싱 (Dual-Cadence Gradual Rebalancing)

### 5.1 설계 배경 및 파라미터

Contest 대회 규정상 **매주 최소 5% 이상의 편도 회전율**이 요구된다.
단순 주간 전량 리밸런싱은 200% 이상의 월간 회전율을 유발해 거래비용이 과다해진다.
이를 해결하기 위해 매주 5~6% 범위 내에서 목표 포트폴리오를 향해 점진적으로 이동한다.

| 파라미터 | 값 | 역할 |
|---|---|---|
| `frequency` | `weekly` | 신호(팩터 스코어) 재계산 주기 |
| `execution_cadence` | `weekly` | 실제 거래 실행 주기 |
| `min_turnover_per_rebalance` | 5% | 주간 최소 편도 회전율 (대회 규정 준수) |
| `max_turnover_per_rebalance` | 6% | 주간 최대 편도 회전율 (거래비용 상한) |

### 5.2 `_partial_target()` 알고리즘

**함수 위치**: `backtest_engine/execution/simulator.py`

**Step 1 — 잔여 갭 계산**

$$\text{needed} = \frac{1}{2} \sum_{i} |w^*_i - w_{t,i}|$$

현재 포트폴리오에서 목표까지 가기 위한 편도(one-way) 회전율.

**Step 2 — 실행 step 결정**

$$\text{step} = \text{clamp}(\text{needed},\ T_{\min},\ T_{\max})$$

단, $\text{step} \leq \text{needed}$ 항상 보장 (목표 초과 방지).

- $\text{needed} < T_{\min}$: 갭이 작으면 갭 전체를 소진 (인위적 과잉 거래 방지)
- $T_{\min} \leq \text{needed} \leq T_{\max}$: 갭 전체 소진
- $\text{needed} > T_{\max}$: $T_{\max}$만큼만 이동, 잔여 갭은 다음 주로 이월

**Step 3 — 중간 목표 비중 산출**

$$\text{scale} = \frac{\text{step}}{\text{needed}}, \qquad w_{t+1} = w_t + (w^* - w_t) \times \text{scale}$$

현재와 목표 사이를 `scale` 비율로 선형 보간한 중간 지점을 당주 실행 목표로 삼는다.

**효과 검증** (실측값 기준)

| 조건 | 평균 월간 회전율 |
|------|----------------|
| 제약 없는 단순 주간 리밸런싱 | ~150~280%/월 |
| **이 전략 (min=5%, max=6%)** | **27.84%/월** |

→ 5-6% × 4주 = 20~24%/월 이론값에 근사하며, 대회 규정(주간 최소 5%)을 준수한다.

### 5.3 엔진 실행 흐름

```python
for trade_date in trading_days:
    # 매일: Mark-to-Market (NAV, 벤치마크 NAV 갱신)

    if trade_date in signal_set:       # 주간 신호일
        pending_target = _compute_target_weights(...)
        # → node_graph(ROE z + vol_neg z) 실행
        # → top_n=20 선택 → equal_weight → 제약 적용
        # 거래 없음 — 목표 비중만 저장

```
